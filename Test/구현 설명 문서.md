# 구현 설명 문서

## 프로젝트 개요

Firebase Realtime Database를 이용하여 사용자별 닉네임, 코인, 점수, 유닛 보유 상태, 아이템 인벤토리를 저장하고 불러오는 구조로 구현하였다.

기본 과제 기능인 회원가입, 로그인, 아이템 구매, 인벤토리 조회 및 사용 기능을 구현하였고, 심화 기능으로 유닛 구매, 게임 결과 저장, 거래소 기능을 추가하였다.

## 1. 구현한 기능 목록

### 기본 기능

| 구분 | 핵심 내용 |
| --- | --- |
| 회원가입 | 저장 경로: `UserInfo/{UserKey}` |
| 회원가입 | 기본값: `Coin = 500`, `Score = 0` |
| 회원가입 | 기본 유닛: `Unit1 = true`, `Unit2~Unit4 = false` |
| 회원가입 | 기본 인벤토리: `HealthPack = 0`, `BarrierCore = 0`, `FlameRune = 0` |
| 닉네임 중복 검사 | `NickName` 기준 검색, 중복 시 가입 차단 |
| 로그인 | 닉네임으로 `UserInfo` 조회 |
| 로그인 | 성공 시 `PlayerPrefs`에 `UserKey`, `UserNickName` 저장 |
| 로그인 | 성공 후 `ShopScene` 이동 |

### 아이템 기능

| 구분 | 핵심 내용 |
| --- | --- |
| 상점 | 구매 아이템: `HealthPack`, `BarrierCore`, `FlameRune` |
| 상점 | 구매 성공: `Coin` 차감, `Inventory` 수량 증가 |
| 상점 | 코인 부족: 구매 차단 |
| 인벤토리 | Firebase `Inventory` 로드 |
| 인벤토리 | 아이템 사용: 수량 1 감소 |
| 인벤토리 | 수량 0개: 사용 차단 |

### 심화 기능

| 구분 | 핵심 내용 |
| --- | --- |
| 유닛 구매 | 구매 대상: `Unit2`, `Unit3`, `Unit4` |
| 유닛 구매 | 구매 성공: `Coin` 차감, `UnitList` 값 `true` 변경 |
| 유닛 구매 | 보유 유닛: 재구매 차단 |
| 게임 결과 | 전투 기준: 보유한 유닛 중 가장 높은 단계 |
| 게임 결과 | 전투 조건: 연습 `Unit1`, 일반 `Unit2`, 보스 `Unit3` 이상 |
| 게임 결과 | 결과 저장: `Coin` 증가, `Score`는 최고 기록일 때만 갱신 |
| 거래소 | 판매 등록: 보유 아이템 수량 1 감소, `AuctionList` 생성 |
| 거래소 | 구매 처리: 구매자 `Coin` 차감, 구매자 `Inventory` 증가 |
| 거래소 | 판매자 처리: 판매자 `Coin` 증가 |
| 거래소 | 판매 완료: `IsSold = true` |

### 공통 처리

| 구분 | 핵심 내용 |
| --- | --- |
| Firebase 콜백 | UI 갱신 시 `UnityMainThreadDispatcher` 사용 |
| Firebase 참조 | `FirebaseDatabaseProvider`로 DatabaseReference 생성 방식 통일 |

## 4. Firebase 데이터 구조 설명

사용자 데이터는 `UserInfo` 경로 아래에 사용자별 고유 키를 기준으로 저장한다.

### 사용자 데이터 구조

```text
UserInfo / {UserKey}
├─ NickName  : 사용자 닉네임
├─ Coin      : 현재 보유 코인
├─ Score     : 최고 점수
├─ UnitList  : 유닛 보유 상태를 저장한 JSON 문자열
└─ Inventory : 아이템 보유 수량을 저장한 JSON 문자열
```

회원가입 직후 사용자 데이터 예시는 다음과 같다.

```json
{
  "NickName": "사용자닉네임",
  "Coin": 500,
  "Score": 0,
  "UnitList": "{\"Unit1\":true,\"Unit2\":false,\"Unit3\":false,\"Unit4\":false}",
  "Inventory": "{\"HealthPack\":0,\"BarrierCore\":0,\"FlameRune\":0}"
}
```

거래소 데이터는 `AuctionList` 경로 아래에 판매 등록별 고유 키를 기준으로 저장한다.

### 거래소 데이터 구조

```text
AuctionList / {ListingKey}
├─ SellerKey      : 판매자 UserKey
├─ SellerNickName : 판매자 닉네임
├─ ItemName       : 판매 아이템 이름
├─ Price          : 판매 가격
├─ IsSold         : 판매 완료 여부
├─ CreatedAt      : 판매 등록 시간
├─ BuyerKey       : 구매자 UserKey
└─ SoldAt         : 판매 완료 시간
```

판매 등록 데이터 예시는 다음과 같다.

```json
{
  "SellerKey": "판매자 UserKey",
  "SellerNickName": "판매자 닉네임",
  "ItemName": "HealthPack",
  "Price": 80,
  "IsSold": false,
  "CreatedAt": "ServerValue.Timestamp"
}
```

닉네임 중복 검사와 닉네임 로그인을 위해 Firebase Rules에는 `NickName` 인덱스를 추가하였다.

```json
{
  "rules": {
    ".read": true,
    ".write": true,
    "UserInfo": {
      ".indexOn": ["NickName"]
    }
  }
}
```

## 5. `PlayerPrefs`에 저장한 값 설명

| 키 | 저장 값 | 사용 목적 |
| --- | --- | --- |
| `UserKey` | Firebase `UserInfo` 아래에 생성된 사용자 고유 키 | 로그인 이후 모든 씬에서 현재 사용자의 Firebase 데이터를 찾는 기준값 |
| `UserNickName` | 로그인 또는 회원가입에 성공한 사용자의 닉네임 | 현재 사용자 표시, 거래소 판매자 닉네임 표시, 거래소 판매 등록에 사용 |

로그아웃 시에는 `UserKey`와 `UserNickName`을 삭제하여 다른 계정으로 로그인할 때 이전 사용자 정보가 남지 않도록 처리하였다.

## 6. `Inventory` JSON 처리 방식 설명

`Inventory`는 C# 코드에서 `Dictionary<string, int>` 형태로 관리한다.

- key: 아이템 이름
- value: 해당 아이템의 보유 수량

관리하는 아이템 key는 다음과 같다.

- `HealthPack`
- `BarrierCore`
- `FlameRune`

Firebase에는 `Dictionary`를 그대로 저장하지 않고 JSON 문자열로 변환하여 저장하였다. 변환에는 Newtonsoft.Json의 `JsonConvert.SerializeObject()`를 사용하였다.

저장되는 `Inventory` JSON 예시는 다음과 같다.

```json
{
  "HealthPack": 1,
  "BarrierCore": 0,
  "FlameRune": 2
}
```

Firebase에서 데이터를 불러올 때는 `JsonConvert.DeserializeObject<Dictionary<string, int>>()`를 사용하여 JSON 문자열을 다시 `Dictionary`로 복원한다.

### 아이템 구매 처리

- 현재 `Inventory` JSON을 `Dictionary`로 변환한다.
- 구매한 아이템의 수량을 1 증가시킨다.
- 변경된 `Dictionary`를 다시 JSON 문자열로 변환한다.
- Firebase의 `Inventory` 값에 저장한다.

### 아이템 사용 및 거래소 판매 등록 처리

- 현재 `Inventory` JSON을 `Dictionary`로 변환한다.
- 해당 아이템 수량이 1 이상인지 먼저 검사한다.
- 수량이 충분하면 1 감소시킨다.
- 변경된 `Dictionary`를 다시 JSON 문자열로 변환하여 Firebase에 저장한다.

## 7. 실행 중 발생한 문제와 해결 방법

| 발생한 문제 | 원인 | 해결 방법 |
| --- | --- | --- |
| `UnityMainThreadDispatcher could not find the UnityMainThreadDispatcher object` 오류 | Firebase 콜백 이후 UI를 갱신하는 스크립트가 씬 안의 Dispatcher 오브젝트를 찾지 못함 | 필요한 씬에 `UnityMainThreadDispatcher` 오브젝트를 배치하고, Firebase 콜백 이후 UI 갱신은 `Enqueue()`로 처리 |
| 회원가입 후 같은 닉네임이 중복 생성됨 | 닉네임 중복 검사 쿼리가 Firebase Rules 인덱스 없이 동작하면서 기존 닉네임을 정상적으로 찾지 못함 | `UserInfo`에 `".indexOn": ["NickName"]` 규칙 추가, 쿼리 실패 시 전체 스캔으로 한 번 더 확인하도록 보완 |
| 회원가입한 닉네임으로 로그인 시 `존재하지 않는 닉네임` 메시지 출력 | 로그인 쿼리도 `NickName` 기준 검색을 사용하므로 Firebase Rules 인덱스가 없을 때 조회가 실패함 | Firebase Rules에 `NickName` 인덱스 추가 후 로그인 쿼리 정상화 |
| 닉네임 입력 중 `ArgumentOutOfRangeException: startIndex cannot be larger than length of string` 발생 | Unity Legacy `InputField`에서 한글 입력 조합 중 커서 위치와 문자열 길이가 어긋남 | `SafeInputField`를 적용하여 입력 필드 선택 위치를 보정 |
| 거래소 테스트 중 본인이 등록한 판매글도 구매 대상에 포함됨 | 판매자와 구매자를 구분하는 `SellerKey` 비교가 부족함 | 구매 전 `SellerKey`와 현재 사용자의 `UserKey`를 비교하여 자기 아이템 구매 차단 |
| 심화 기능 화면의 텍스트가 영어로 표시됨 | 초기 구성 과정에서 임시 UI 문구가 남아 있음 | 버튼, 안내 문구, 상태 메시지를 한국어로 수정 |
| 심화 기능 구현 후 콘솔 오류가 여러 개 누적됨 | 심화 기능을 임시 구성하면서 씬 참조, 입력 처리, UI 연결이 섞임 | 기본 과제와 같은 방식으로 씬별 Manager 스크립트와 Inspector `OnClick` 직접 연결 방식으로 재구성 |
